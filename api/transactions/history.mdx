## Overview

This endpoint provides access to the authenticated user's transaction history with pagination support. It also supports updating transaction status via PATCH requests.

## Authentication

Requires a valid JWT token in the `Authorization` header:

```
Authorization: Bearer <token>
```

## GET - List Transactions

Retrieves the user's transaction history, ordered by newest first.

### Query Parameters

<ParamField query="limit" type="number" default="20">
  Number of transactions to return (max: 100)
</ParamField>

<ParamField query="offset" type="number" default="0">
  Number of transactions to skip for pagination
</ParamField>

### Response

Returns an array of transaction objects:

<ResponseField name="id" type="number">
  Unique transaction ID
</ResponseField>

<ResponseField name="userId" type="number">
  ID of the user who owns this transaction
</ResponseField>

<ResponseField name="hash" type="string">
  Transaction hash on the blockchain
</ResponseField>

<ResponseField name="chainId" type="number">
  EVM chain ID (1 = Ethereum, 137 = Polygon, etc.)
</ResponseField>

<ResponseField name="chainType" type="string">
  Blockchain type (currently only "EVM")
</ResponseField>

<ResponseField name="fromAddress" type="string">
  Sender's wallet address
</ResponseField>

<ResponseField name="toAddress" type="string">
  Recipient's wallet address
</ResponseField>

<ResponseField name="tokenAddress" type="string | null">
  ERC-20 token contract address (null for native tokens)
</ResponseField>

<ResponseField name="tokenSymbol" type="string">
  Token symbol (e.g., "ETH", "USDC", "MATIC")
</ResponseField>

<ResponseField name="value" type="string">
  Transaction amount in token's smallest unit
</ResponseField>

<ResponseField name="status" type="string">
  Transaction status: `"pending"`, `"confirmed"`, or `"failed"`
</ResponseField>

<ResponseField name="gasUsed" type="string | null">
  Gas consumed by the transaction (populated after confirmation)
</ResponseField>

<ResponseField name="blockNumber" type="string | null">
  Block number where transaction was mined
</ResponseField>

<ResponseField name="createdAt" type="string">
  ISO 8601 timestamp when transaction was recorded
</ResponseField>

### Error Responses

- `401` - Unauthorized (invalid or missing token)

### Example Request

<CodeGroup>

```bash cURL
curl -X GET "https://api.walty.app/api/tx?limit=10&offset=0" \
  -H "Authorization: Bearer <your-token>"
```

## PATCH - Update Transaction Status

Updates the status of an existing transaction.

### Request Body

<ParamField body="hash" type="string" required>
  Transaction hash to update
</ParamField>

<ParamField body="status" type="string" required>
  New status: `"confirmed"` or `"failed"`
</ParamField>

### Response

<ResponseField name="ok" type="boolean">
  Returns `true` if the update was successful
</ResponseField>

### Error Responses

<ResponseField name="error" type="string">
  Error message describing what went wrong
</ResponseField>

- `400` - Invalid payload (missing hash or invalid status)
- `401` - Unauthorized (invalid or missing token)

<Warning>
You can only update transactions that belong to your user account. The status can only be changed to `"confirmed"` or `"failed"`.
</Warning>

### Example Request

<CodeGroup>

```bash cURL
curl -X PATCH https://api.walty.app/api/tx \
  -H "Authorization: Bearer <your-token>" \
  -H "Content-Type: application/json" \
  -d '{
    "hash": "0x1234567890abcdef1234567890abcdef1234567890abcdef1234567890abcdef",
    "status": "confirmed"
  }'
```

```javascript JavaScript
const response = await fetch('https://api.walty.app/api/tx', {
  method: 'PATCH',
  headers: {
    'Authorization': 'Bearer <your-token>',
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    hash: '0x1234567890abcdef1234567890abcdef1234567890abcdef1234567890abcdef',
    status: 'confirmed'
  })
});

const result = await response.json();
```

```python Python
import requests

response = requests.patch(
    'https://api.walty.app/api/tx',
    headers={
        'Authorization': 'Bearer <your-token>',
        'Content-Type': 'application/json'
    },
    json={
        'hash': '0x1234567890abcdef1234567890abcdef1234567890abcdef1234567890abcdef',
        'status': 'confirmed'
    }
)

result = response.json()
```

</CodeGroup>

## Implementation Notes

### Pagination

For efficient pagination through large transaction histories:

```javascript
let offset = 0;
const limit = 50;
let allTransactions = [];

while (true) {
  const batch = await fetchTransactions({ limit, offset });
  if (batch.length === 0) break;
  
  allTransactions.push(...batch);
  offset += limit;
}
```

### Status Lifecycle

1. **pending** - Transaction broadcast to network but not yet mined
2. **confirmed** - Transaction successfully mined and confirmed
3. **failed** - Transaction reverted or failed on-chain

<Note>
Transactions are created with `pending` status via the POST endpoint. Monitor on-chain status and use PATCH to update accordingly.
</Note>

### Transaction Monitoring

Recommended flow for tracking transaction status:

1. Create transaction record with POST `/api/tx`
2. Broadcast transaction to blockchain
3. Poll blockchain for transaction receipt
4. Update status with PATCH `/api/tx` when confirmed/failed
5. Optionally populate `gasUsed` and `blockNumber` fields
