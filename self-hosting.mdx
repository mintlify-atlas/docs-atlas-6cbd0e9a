Walty can be self-hosted on your own infrastructure using Docker. This guide covers production deployment, environment configuration, and operational considerations.

## Why Self-Host?

Self-hosting Walty gives you:

- **Full control** over your wallet infrastructure
- **Privacy** - no tracking or analytics by default
- **Security** - keep sensitive operations within your network
- **Customization** - modify and extend as needed

## Production Deployment

<Steps>
  <Step title="Clone the repository">
    ```bash
    git clone https://github.com/ignaciogarcia-dev/walty.git
    cd walty
    ```
  </Step>

  <Step title="Configure environment variables">
    Copy the environment template and configure for production:

    ```bash
    cp .env.example .env
    ```

    Edit `.env` with your production values. See [Environment Variables](#environment-variables) below for details.

    <Warning>
      Never use the default `JWT_SECRET` or `SERVER_PEPPER` values in production. Generate secure secrets using `openssl rand -base64 32`.
    </Warning>
  </Step>

  <Step title="Build and start services">
    ```bash
    docker compose up --build -d
    ```

    The `-d` flag runs containers in detached mode (background).
  </Step>

  <Step title="Verify deployment">
    Check that all services are running:

    ```bash
    docker compose ps
    ```

    Check application logs:

    ```bash
    docker compose logs -f app
    ```
  </Step>
</Steps>

## Environment Variables

### Required Variables

These variables are essential for Walty to function:

<CodeGroup>

```bash Database
# PostgreSQL database URL
# For Docker Compose, use the 'db' service hostname
DATABASE_URL=postgresql://wallet:wallet@db:5432/wallet

# For external database (recommended for production):
# DATABASE_URL=postgresql://user:password@your-db-host:5432/wallet
```

```bash Secrets
# Secret used to sign and verify JWTs
# Generate: openssl rand -base64 32
JWT_SECRET=your-secret-key-here-change-this-in-production

# HMAC key for deriving the PIN challenge used in server-side wallet backup
# Generate: openssl rand -base64 32
SERVER_PEPPER=your-pepper-key-here-change-this-in-production
```

```bash API Keys
# Alchemy API key for multichain RPC
# Ethereum, Arbitrum, Base, Optimism, Polygon
# Get free key at: https://www.alchemy.com/
ALCHEMY_API_KEY=your-alchemy-api-key

# 0x API key for token swaps
# Get free key at: https://0x.org/docs/api
ZEROX_API_KEY=your-zerox-api-key
```

</CodeGroup>

<Warning>
  Without `ALCHEMY_API_KEY`, balance and transaction queries will fail as public RPCs are unreliable. Without `ZEROX_API_KEY`, swap calls will fail with 401. These are effectively required for full functionality.
</Warning>

### Optional Variables

These variables enhance functionality but are not required:

<CodeGroup>

```bash RPC Providers
# Ankr API key for multichain RPC (fallback between Alchemy and public RPCs)
# Get one at: https://www.ankr.com/rpc/
ANKR_API_KEY=
```

```bash Swap Providers
# 1inch API key for swap fallback
# Get one at: https://portal.1inch.dev/
ONEINCH_API_KEY=
```

```bash Pricing
# CoinGecko API key for token prices and images
# Get one at: https://www.coingecko.com/en/api
COINGECKO_API_KEY=

# Override CoinGecko base URL (rarely needed)
# COINGECKO_API_BASE_URL=https://api.coingecko.com/api/v3
```

```bash Security
# Force the Secure flag on the JWT cookie
# Useful for local HTTPS development
# If not set, Secure is only applied in production (NODE_ENV=production)
COOKIE_SECURE=true
```

</CodeGroup>

<Info>
  Without `COINGECKO_API_KEY`, the app works but with stricter rate limits on the free tier.
</Info>

## Docker Compose Configuration

The default `docker-compose.yml` provides a complete stack:

```yaml docker-compose.yml
services:
  app:
    build: .
    ports:
      - "3000:3000"
    env_file:
      - .env
    depends_on:
      - db
    environment:
      - CI=true

  db:
    image: postgres:16
    restart: always
    environment:
      POSTGRES_USER: wallet
      POSTGRES_PASSWORD: wallet
      POSTGRES_DB: wallet
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

### Key Points

- **PostgreSQL 16** is used for the database
- **Volume persistence** ensures database data survives container restarts
- **Automatic restarts** for the database service
- **Environment variables** loaded from `.env` file

## Database Migrations

Walty uses Drizzle ORM for database management. Migrations run automatically when the app container starts.

### Automatic Migration Flow

The `entrypoint.sh` script handles the migration process:

```bash entrypoint.sh
#!/bin/sh
set -e

echo "Waiting for database..."

# Wait for PostgreSQL to be ready
until node -e "
const { Client } = require('pg');
const client = new Client({ connectionString: process.env.DATABASE_URL });
client.connect()
  .then(() => client.query('SELECT 1'))
  .then(() => client.end())
  .then(() => process.exit(0))
  .catch(() => process.exit(1));
"; do
  sleep 1
done

echo "Running DB migrations..."
if pnpm drizzle-kit push; then
  echo "✓ Database schema synchronized"
  echo "Database ready"
else
  echo "✗ Migration failed"
  exit 1
fi

echo "Starting app..."
exec pnpm start
```

### Manual Migration

If you need to run migrations manually:

```bash
# Inside the app container
docker compose exec app pnpm db:migrate

# Or using Drizzle Studio (database GUI)
docker compose exec app pnpm db:studio
```

## Production Considerations

### Security

<Steps>
  <Step title="Generate secure secrets">
    Always generate new secrets for production:

    ```bash
    # Generate JWT_SECRET
    openssl rand -base64 32

    # Generate SERVER_PEPPER
    openssl rand -base64 32
    ```
  </Step>

  <Step title="Use HTTPS in production">
    Set `COOKIE_SECURE=true` to enforce secure cookies:

    ```bash
    COOKIE_SECURE=true
    ```

    Deploy behind a reverse proxy (nginx, Caddy) with SSL/TLS.
  </Step>

  <Step title="Restrict database access">
    Use a dedicated database user with limited permissions:

    ```bash
    DATABASE_URL=postgresql://walty_user:secure_password@db:5432/walty_prod
    ```
  </Step>

  <Step title="API key security">
    Store API keys securely and never commit them to version control. Consider using secrets management solutions like:
    - Docker secrets
    - HashiCorp Vault
    - Cloud provider secret managers (AWS Secrets Manager, GCP Secret Manager)
  </Step>
</Steps>

### Scalability

<CardGroup cols={2}>
  <Card title="External Database" icon="database">
    For production, use a managed PostgreSQL instance (AWS RDS, Google Cloud SQL, etc.) instead of the containerized database.
  </Card>
  <Card title="Load Balancing" icon="scale-balanced">
    Run multiple app containers behind a load balancer for high availability.
  </Card>
  <Card title="Caching" icon="bolt">
    Consider adding Redis for session caching and rate limiting.
  </Card>
  <Card title="Monitoring" icon="chart-line">
    Implement monitoring and logging (Prometheus, Grafana, etc.).
  </Card>
</CardGroup>

### Database Backup

Regularly backup your PostgreSQL database:

```bash
# Backup
docker compose exec db pg_dump -U wallet wallet > backup_$(date +%Y%m%d_%H%M%S).sql

# Restore
cat backup_20260313_120000.sql | docker compose exec -T db psql -U wallet wallet
```

For production, automate backups and store them securely offsite.

### Resource Limits

Set resource limits in your `docker-compose.yml`:

```yaml docker-compose.yml
services:
  app:
    deploy:
      resources:
        limits:
          cpus: '2'
          memory: 2G
        reservations:
          cpus: '1'
          memory: 1G

  db:
    deploy:
      resources:
        limits:
          cpus: '2'
          memory: 4G
        reservations:
          cpus: '1'
          memory: 2G
```

## Reverse Proxy Setup

### Nginx Example

```nginx nginx.conf
server {
    listen 80;
    server_name wallet.example.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl http2;
    server_name wallet.example.com;

    ssl_certificate /etc/ssl/certs/wallet.example.com.crt;
    ssl_certificate_key /etc/ssl/private/wallet.example.com.key;

    location / {
        proxy_pass http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection 'upgrade';
        proxy_set_header Host $host;
        proxy_cache_bypass $http_upgrade;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### Caddy Example

```caddyfile Caddyfile
wallet.example.com {
    reverse_proxy localhost:3000
}
```

Caddy automatically handles SSL/TLS certificates via Let's Encrypt.

## Monitoring and Logging

### View Logs

```bash
# All services
docker compose logs -f

# App only
docker compose logs -f app

# Database only
docker compose logs -f db

# Last 100 lines
docker compose logs --tail=100 app
```

### Health Checks

Add health checks to `docker-compose.yml`:

```yaml docker-compose.yml
services:
  app:
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/api/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s

  db:
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U wallet"]
      interval: 10s
      timeout: 5s
      retries: 5
```

## Troubleshooting

### Container Won't Start

Check logs for errors:

```bash
docker compose logs app
```

Common issues:
- Missing or invalid environment variables
- Database connection failure
- Port conflicts

### Database Connection Issues

1. Verify database is running:
   ```bash
   docker compose ps db
   ```

2. Test database connectivity:
   ```bash
   docker compose exec db psql -U wallet -d wallet -c "SELECT 1;"
   ```

3. Check `DATABASE_URL` format:
   ```bash
   postgresql://user:password@host:port/database
   ```

### Migration Failures

If migrations fail on startup:

1. Check migration logs:
   ```bash
   docker compose logs app | grep migration
   ```

2. Run migrations manually:
   ```bash
   docker compose exec app pnpm db:migrate
   ```

3. Access Drizzle Studio for debugging:
   ```bash
   docker compose exec app pnpm db:studio
   ```

### Performance Issues

If the app is slow:

1. Check resource usage:
   ```bash
   docker stats
   ```

2. Review database query performance:
   ```bash
   docker compose exec app pnpm db:studio
   ```

3. Consider scaling horizontally with multiple app instances
